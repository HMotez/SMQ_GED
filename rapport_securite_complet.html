🚨 Vulnérabilité Critique non couverte par l'audit initial
Le fallback x-user-id dans roleMiddleware.js:70 permet à n'importe quel client d'envoyer un header HTTP x-user-id: 1 et d'agir en tant qu'administrateur sans token JWT. Cette faille contourne totalement le système d'authentification.

Plan de correction — Priorités
1
CRITIQUE   Supprimer le fallback x-user-id (contournement JWT)
Supprimer les lignes 70–88 de roleMiddleware.js. Ce fallback permet à n'importe quel client d'envoyer x-user-id: 1 et d'agir en tant qu'administrateur sans token JWT valide.
2
CRITIQUE   Protection Brute Force — Rate Limiting
Installer express-rate-limit. Appliquer sur POST /api/auth/login : max 3 tentatives par fenêtre de 15 min. Bloquer le compte après dépassement. Fichier : authRoutes.js + authController.js:116
3
CRITIQUE   Invalidation du JWT à la déconnexion
Implémenter une blacklist de tokens en Redis ou en base de données. La fonction logout() doit stocker le JTI du token. Le middleware doit vérifier la blacklist avant d'autoriser. Fichier : authController.js:301
4
CRITIQUE   Configurer HTTPS / SSL en production
Configurer Let's Encrypt/Certbot dans Nginx. Ajouter un bloc server { listen 443 ssl; } et rediriger le port 80 → 443. Fichier : nginx.conf, docker-compose.yml
5
IMPORTANT   Ajouter helmet.js — Headers de sécurité HTTP
npm install helmet puis app.use(helmet()) dans server.js. Ajoute automatiquement : CSP, X-XSS-Protection, HSTS, X-Frame-Options, X-Content-Type-Options, supprime X-Powered-By.
6
IMPORTANT   Restreindre la politique CORS
Remplacer app.use(cors()) par app.use(cors({ origin: process.env.ALLOWED_ORIGINS?.split(',') })) avec la liste des domaines autorisés dans .env. Fichier : server.js:20
7
IMPORTANT   Renforcer la politique de mots de passe
Exiger minimum 8 caractères avec au moins 1 majuscule, 1 chiffre et 1 caractère spécial. Appliquer via regex dans authController.js:249 et authController.js:401. Externaliser les mots de passe des seeds via .env.
8
IMPORTANT   Ne pas exposer le port backend 4000 publiquement
Supprimer ports: "4000:4000" du service backend dans docker-compose.yml:88. L'API doit être accessible uniquement via le reverse proxy Nginx sur port 80/443.
9
IMPORTANT   Masquer les versions des technologies
Ajouter server_tokens off; dans nginx.conf. Helmet.js (recommandation #5) supprime automatiquement X-Powered-By: Express. Cela empêche la reconnaissance technologique passive.
10
MOYEN   Masquer les messages d'erreur internes
Remplacer res.json({ error: err.message }) par res.json({ error: "Erreur serveur." }) dans server.js:190. Logger l'erreur complète en interne uniquement. Idem pour les routes de conversion.
11
MOYEN   Créer un utilisateur PostgreSQL dédié (non superuser)
Créer un utilisateur smq_app avec uniquement les droits CRUD nécessaires. Remplacer postgres dans .env. Principe du moindre privilège appliqué à la base de données.
12
MOYEN   Externaliser les credentials des comptes par défaut
Déplacer les mots de passe des seeds (Admin123!, etc.) vers des variables d'environnement dans .env. Ne jamais coder en dur des credentials dans le code source versionné. Fichier : authController.js:20-24
13
FAIBLE   Mettre en place une procédure de sauvegarde automatisée
Configurer un cron pg_dump quotidien avec stockage distant. Documenter les procédures de PRA/PCA. Tester la restauration périodiquement.
14
FAIBLE   Implémenter un monitoring applicatif
Intégrer Prometheus + Grafana via docker-compose.yml pour superviser les métriques (CPU, mémoire, temps de réponse, erreurs 5xx). Configurer des alertes automatiques.
15
FAIBLE   Activer npm audit dans le CI/CD
Exécuter npm audit --audit-level=high à chaque build pour détecter les dépendances vulnérables. Intégrer Snyk ou Dependabot pour les alertes automatiques sur les CVE.
